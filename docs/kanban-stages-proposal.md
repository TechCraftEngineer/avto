# Предложение: новые колонки Kanban для откликов

## Проблема

Текущие колонки основаны на `response.status` (внутренние этапы AI-скрининга):

| Сейчас | Проблема |
|--------|----------|
| Новые | Внутренний статус системы, непонятно рекрутеру |
| Оценённые | «AI оценил» — технический термин |
| Собеседование | Норм |
| Переговоры | Норм |
| Завершённые | Слишком размыто (что завершено?) |
| Пропущенные | Норм, но в отдельной колонке |

## Решение: stage-based колонки (как у кандидатов)

Используем уже существующую логику **воронки рекрутмента** (`FunnelStage`) — она маппит `status` + `hrSelectionStatus` в понятные этапы.

### Предлагаемые колонки

| Stage ID | Название | Кто попадает | Понятность рекрутеру |
|----------|----------|--------------|----------------------|
| **SCREENING_DONE** | На рассмотрении | NEW, EVALUATED (без hrSelectionStatus) | Кандидаты на первичный просмотр |
| **INTERVIEW** | Собеседование | INTERVIEW, NEGOTIATION | Идёт диалог/чат с кандидатом |
| **OFFER_SENT** | Оффер | hrSelectionStatus: OFFER, INVITE, RECOMMENDED, SELECTED | Готовы к приглашению / оффер отправлен |
| **SECURITY_PASSED** | СБ пройдена | hrSelectionStatus: SECURITY_PASSED | Проверка безопасности |
| **CONTRACT_SENT** | Договор отправлен | hrSelectionStatus: CONTRACT_SENT | Подписание |
| **ONBOARDING** | Онбординг | hrSelectionStatus: ONBOARDING | Адаптация |
| **REJECTED** | Отказ | REJECTED, NOT_RECOMMENDED, SKIPPED, COMPLETED (закрыт без найма) | Не подошёл / отклонён |

### Сравнение

**Было (6 колонок по status):**
- Новые → Оценённые → Собеседование → Переговоры → Завершённые | Пропущенные

**Станет (7 колонок по stage):**
- На рассмотрении → Собеседование → Оффер → СБ пройдена → Договор отправлен → Онбординг | Отказ

### Единообразие

Pipeline кандидатов уже использует эти же стадии. Отклики (ResponsesKanban) будут визуально и концептуально совпадать с воронкой кандидатов — рекрутер видит один и тот же язык этапов.

### API

- **Обновление при drag**: `candidates.updateStage` (уже реализован) — обновляет `status` и `hrSelectionStatus` по stage
- **Группировка**: маппинг `status` + `hrSelectionStatus` → stage на клиенте (как в candidates.list)
