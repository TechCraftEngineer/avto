# Требования к функционалу импорта вакансий

## Введение

Система импорта вакансий предназначена для рекрутеров, которым необходимо загружать вакансии из внешних источников (например, hh.ru) в систему. Функционал поддерживает как массовый импорт новых и архивных вакансий, так и импорт отдельных вакансий по ссылке. Все операции выполняются асинхронно через Inngest с отображением прогресса в реальном времени.

## Глоссарий

- **Система**: Система управления вакансиями для рекрутеров
- **Импорт**: Процесс загрузки данных о вакансиях из внешнего источника в систему
- **Новая_Вакансия**: Вакансия, которая активна на внешнем источнике и не была ранее импортирована
- **Архивная_Вакансия**: Вакансия, которая была закрыта или завершена на внешнем источнике
- **Inngest_Задание**: Фоновая задача, выполняемая через Inngest
- **Прогресс_Бар**: Визуальный индикатор выполнения операции в реальном времени
- **Токен_Подписки**: Токен для подключения к Inngest Realtime каналу
- **Рабочее_Пространство**: Изолированная область данных для организации или команды

## Требования

### Требование 1: Массовый импорт новых вакансий

**Пользовательская история:** Как рекрутер, я хочу импортировать все мои активные вакансии из внешнего источника, чтобы начать работу с ними в системе.

#### Критерии приемки

1. WHEN рекрутер запускает массовый импорт новых вакансий, THEN Система SHALL создать Inngest_Задание для импорта
2. WHEN Inngest_Задание начинает выполнение, THEN Система SHALL получить список всех активных вакансий из внешнего источника
3. WHEN Система получает список вакансий, THEN Система SHALL сохранить каждую Новую_Вакансию в базу данных
4. WHEN Система сохраняет вакансию, THEN Система SHALL сохранить все основные поля вакансии (название, описание, зарплата, требования)
5. WHEN импорт завершается, THEN Система SHALL вернуть количество успешно импортированных вакансий

### Требование 2: Массовый импорт архивных вакансий

**Пользовательская история:** Как рекрутер, я хочу импортировать мои закрытые вакансии из внешнего источника, чтобы иметь полную историю работы.

#### Критерии приемки

1. WHEN рекрутер запускает массовый импорт архивных вакансий, THEN Система SHALL создать Inngest_Задание для импорта
2. WHEN Inngest_Задание начинает выполнение, THEN Система SHALL получить список всех архивных вакансий из внешнего источника
3. WHEN Система получает список вакансий, THEN Система SHALL сохранить каждую Архивную_Вакансию в базу данных с отметкой о статусе
4. WHEN Система сохраняет архивную вакансию, THEN Система SHALL сохранить все основные поля и дату закрытия
5. WHEN импорт завершается, THEN Система SHALL вернуть количество успешно импортированных архивных вакансий

### Требование 3: Импорт одной вакансии по ссылке

**Пользовательская история:** Как рекрутер, я хочу импортировать конкретную вакансию по её ссылке, чтобы быстро добавить её в систему.

#### Критерии приемки

1. WHEN рекрутер вводит ссылку на вакансию и запускает импорт, THEN Система SHALL валидировать формат ссылки
2. WHEN ссылка валидна, THEN Система SHALL создать Inngest_Задание для импорта одной вакансии
3. WHEN Inngest_Задание начинает выполнение, THEN Система SHALL извлечь данные вакансии по ссылке
4. WHEN данные получены, THEN Система SHALL сохранить вакансию в базу данных
5. IF ссылка невалидна, THEN Система SHALL вернуть сообщение об ошибке с описанием проблемы

### Требование 4: Отображение прогресса в реальном времени

**Пользовательская история:** Как рекрутер, я хочу видеть прогресс импорта в реальном времени, чтобы понимать, сколько времени займёт операция.

#### Критерии приемки

1. WHEN рекрутер запускает любую операцию импорта, THEN Система SHALL получить Токен_Подписки для Inngest Realtime канала
2. WHEN Токен_Подписки получен, THEN Система SHALL подключиться к каналу прогресса
3. WHEN Inngest_Задание обновляет прогресс, THEN Система SHALL отправить событие прогресса в канал
4. WHEN событие прогресса получено, THEN Система SHALL обновить Прогресс_Бар с текущим состоянием
5. WHEN импорт завершается, THEN Система SHALL отобразить финальный результат с количеством импортированных вакансий

### Требование 5: Обработка ошибок импорта

**Пользовательская история:** Как рекрутер, я хочу получать понятные сообщения об ошибках, чтобы понимать, что пошло не так при импорте.

#### Критерии приемки

1. IF внешний источник недоступен, THEN Система SHALL вернуть сообщение "Не удалось подключиться к источнику вакансий"
2. IF учётные данные для доступа невалидны, THEN Система SHALL вернуть сообщение "Проверьте настройки доступа к источнику"
3. IF вакансия уже существует в системе, THEN Система SHALL пропустить её и продолжить импорт остальных
4. IF произошла ошибка при сохранении вакансии, THEN Система SHALL записать ошибку в лог и продолжить импорт
5. WHEN импорт завершается с ошибками, THEN Система SHALL отобразить количество успешных и неуспешных операций

### Требование 6: Валидация данных вакансий

**Пользовательская история:** Как рекрутер, я хочу быть уверенным, что импортированные вакансии содержат корректные данные.

#### Критерии приемки

1. WHEN Система получает данные вакансии, THEN Система SHALL валидировать обязательные поля (название, описание)
2. IF обязательное поле отсутствует, THEN Система SHALL пропустить вакансию и записать ошибку валидации
3. WHEN Система валидирует зарплату, THEN Система SHALL проверить, что значения являются числами
4. WHEN Система валидирует даты, THEN Система SHALL проверить корректность формата дат
5. WHEN все данные валидны, THEN Система SHALL сохранить вакансию в базу данных

### Требование 7: Изоляция данных по рабочим пространствам

**Пользовательская история:** Как рекрутер, я хочу, чтобы мои импортированные вакансии были видны только в моём рабочем пространстве.

#### Критерии приемки

1. WHEN рекрутер запускает импорт, THEN Система SHALL определить текущее Рабочее_Пространство пользователя
2. WHEN Система сохраняет вакансию, THEN Система SHALL связать вакансию с Рабочим_Пространством
3. WHEN рекрутер просматривает импортированные вакансии, THEN Система SHALL отображать только вакансии его Рабочего_Пространства
4. IF пользователь не имеет доступа к Рабочему_Пространству, THEN Система SHALL запретить импорт вакансий
5. WHEN импорт завершается, THEN Система SHALL подтвердить, что все вакансии сохранены в правильном Рабочем_Пространстве

### Требование 8: Предотвращение дублирования вакансий

**Пользовательская история:** Как рекрутер, я хочу избежать дублирования вакансий при повторном импорте.

#### Критерии приемки

1. WHEN Система импортирует вакансию, THEN Система SHALL проверить наличие вакансии с таким же внешним идентификатором
2. IF вакансия уже существует, THEN Система SHALL обновить существующую вакансию вместо создания новой
3. WHEN Система обновляет вакансию, THEN Система SHALL сохранить все новые данные
4. WHEN Система обновляет вакансию, THEN Система SHALL сохранить дату последнего обновления
5. WHEN импорт завершается, THEN Система SHALL вернуть количество новых и обновлённых вакансий

### Требование 9: Понятный интерфейс для рекрутеров

**Пользовательская история:** Как рекрутер без технических знаний, я хочу легко понимать интерфейс импорта вакансий.

#### Критерии приемки

1. WHEN рекрутер открывает раздел импорта, THEN Система SHALL отображать три понятные кнопки действий
2. WHEN Система отображает кнопки, THEN Система SHALL использовать простые названия без технических терминов
3. WHEN Система отображает прогресс, THEN Система SHALL показывать процент выполнения и понятное описание текущего действия
4. WHEN Система отображает результат, THEN Система SHALL использовать простой язык (например, "Загружено 15 вакансий")
5. IF произошла ошибка, THEN Система SHALL отображать сообщение на понятном языке без технических деталей

### Требование 10: Производительность импорта

**Пользовательская история:** Как рекрутер, я хочу, чтобы импорт выполнялся быстро, даже при большом количестве вакансий.

#### Критерии приемки

1. WHEN Система импортирует вакансии, THEN Система SHALL обрабатывать их пакетами для оптимизации производительности
2. WHEN Система сохраняет вакансии, THEN Система SHALL использовать пакетные операции вставки в базу данных
3. WHEN импорт выполняется, THEN Система SHALL обновлять прогресс не чаще одного раза в секунду
4. WHEN Система получает данные из внешнего источника, THEN Система SHALL использовать параллельные запросы где возможно
5. WHEN импорт большого количества вакансий (более 100), THEN Система SHALL завершить операцию за разумное время
