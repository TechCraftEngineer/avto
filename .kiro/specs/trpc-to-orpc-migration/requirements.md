# Документ требований: Миграция с tRPC на oRPC

## Введение

Данный документ описывает требования к миграции API проекта с tRPC на oRPC. Проект представляет собой Turborepo монорепозиторий с Next.js приложением, использующим tRPC для коммуникации между клиентом и сервером. Миграция должна сохранить всю существующую функциональность, обеспечить типобезопасность и минимизировать breaking changes.

## Глоссарий

- **System**: Система API проекта, включающая серверные роутеры и клиентские вызовы
- **Router**: Роутер API, содержащий набор процедур (endpoints)
- **Procedure**: Отдельная процедура API (query или mutation)
- **Context**: Контекст выполнения процедуры, содержащий сессию, БД и другие зависимости
- **Middleware**: Промежуточное ПО для обработки запросов (аутентификация, rate limiting, логирование)
- **Client**: Клиентская часть приложения, использующая API
- **TanStack_Query**: Библиотека для управления состоянием сервера на клиенте
- **Transformer**: Механизм сериализации/десериализации данных (SuperJSON)
- **Validation**: Валидация входных данных с помощью Zod схем

## Требования

### Требование 1: Миграция серверной конфигурации

**User Story:** Как разработчик, я хочу мигрировать серверную конфигурацию tRPC на oRPC, чтобы использовать новый фреймворк с сохранением всей функциональности.

#### Критерии приемки

1. WHEN система инициализируется, THE System SHALL создавать oRPC контекст с теми же зависимостями (auth, db, repositories, auditLogger, ipAddress, userAgent, interviewToken, inngest, headers)
2. WHEN система конфигурируется, THE System SHALL использовать SuperJSON в качестве трансформера данных
3. WHEN происходит ошибка валидации Zod, THE System SHALL форматировать ошибку с flattenError для клиента
4. THE System SHALL экспортировать тип контекста для использования в процедурах
5. THE System SHALL поддерживать создание роутеров через фабричную функцию

### Требование 2: Миграция middleware

**User Story:** Как разработчик, я хочу мигрировать все существующие middleware на oRPC, чтобы сохранить функциональность безопасности, логирования и rate limiting.

#### Критерии приемки

1. WHEN запрос обрабатывается, THE System SHALL логировать время выполнения процедуры
2. WHEN происходит медленная операция (>5000ms), THE System SHALL логировать предупреждение с деталями
3. WHEN происходит ошибка UNAUTHORIZED, THE System SHALL логировать попытку несанкционированного доступа
4. WHEN происходит ошибка FORBIDDEN, THE System SHALL логировать подозрительную активность
5. WHEN происходит ошибка TOO_MANY_REQUESTS, THE System SHALL логировать превышение rate limit
6. WHEN пользователь выполняет mutation, THE System SHALL логировать событие модификации данных для аудита
7. THE System SHALL применять middleware в правильном порядке (timing → security headers → security audit)

### Требование 3: Миграция процедур (procedures)

**User Story:** Как разработчик, я хочу мигрировать publicProcedure и protectedProcedure на oRPC, чтобы сохранить паттерны авторизации.

#### Критерии приемки

1. THE System SHALL предоставлять publicProcedure для endpoints без требования авторизации
2. THE System SHALL предоставлять protectedProcedure для endpoints с требованием авторизации
3. WHEN используется protectedProcedure и сессия отсутствует, THE System SHALL выбрасывать ошибку UNAUTHORIZED
4. WHEN используется protectedProcedure и сессия существует, THE System SHALL гарантировать наличие ctx.session.user
5. THE System SHALL применять все middleware к обоим типам процедур

### Требование 4: Миграция роутеров

**User Story:** Как разработчик, я хочу мигрировать все 27 существующих роутеров на oRPC, чтобы сохранить структуру API.

#### Критерии приемки

1. THE System SHALL поддерживать вложенные роутеры (workspace.members, workspace.invites)
2. THE System SHALL сохранять структуру экспорта роутеров с использованием satisfies для type-safety
3. THE System SHALL поддерживать условное подключение роутеров (test router только в dev)
4. WHEN роутер экспортируется, THE System SHALL использовать именованные экспорты
5. THE System SHALL сохранять организацию "один файл = одна процедура"

### Требование 5: Миграция валидации входных данных

**User Story:** Как разработчик, я хочу сохранить валидацию входных данных через Zod, чтобы обеспечить типобезопасность.

#### Критерии приемки

1. WHEN процедура определяется, THE System SHALL поддерживать Zod схемы для валидации input
2. WHEN валидация не проходит, THE System SHALL возвращать понятные сообщения об ошибках на русском языке
3. THE System SHALL поддерживать переиспользование схем из пакета @qbs-autonaim/validators
4. THE System SHALL поддерживать inline схемы для простых случаев
5. THE System SHALL использовать Zod v4

### Требование 6: Миграция обработки ошибок

**User Story:** Как разработчик, я хочу сохранить систему обработки ошибок, чтобы клиенты получали понятные сообщения.

#### Критерии приемки

1. THE System SHALL поддерживать стандартные коды ошибок (NOT_FOUND, FORBIDDEN, BAD_REQUEST, UNAUTHORIZED, TOO_MANY_REQUESTS, INTERNAL_SERVER_ERROR)
2. WHEN ресурс не найден, THE System SHALL выбрасывать ошибку с кодом NOT_FOUND
3. WHEN доступ запрещен, THE System SHALL выбрасывать ошибку с кодом FORBIDDEN
4. WHEN данные некорректны, THE System SHALL выбрасывать ошибку с кодом BAD_REQUEST
5. WHEN пользователь не авторизован, THE System SHALL выбрасывать ошибку с кодом UNAUTHORIZED
6. WHEN превышен rate limit, THE System SHALL выбрасывать ошибку с кодом TOO_MANY_REQUESTS с указанием времени до сброса
7. THE System SHALL включать сообщения об ошибках на русском языке

### Требование 7: Миграция клиентской части

**User Story:** Как разработчик, я хочу мигрировать клиентскую часть на oRPC, чтобы сохранить интеграцию с TanStack Query.

#### Критерии приемки

1. THE System SHALL предоставлять React хук для получения типизированного клиента
2. THE System SHALL поддерживать интеграцию с TanStack Query через фабрики queryOptions и mutationOptions
3. THE System SHALL поддерживать использование skipToken для условных запросов
4. THE System SHALL поддерживать useSuspenseQuery для компонентов с Suspense границами
5. THE System SHALL поддерживать server-side prefetch для критичных данных
6. THE System SHALL поддерживать infinite queries
7. THE System SHALL использовать SuperJSON для сериализации данных
8. THE System SHALL поддерживать httpBatchStreamLink для батчинга запросов

### Требование 8: Миграция операций с Query Client

**User Story:** Как разработчик, я хочу сохранить возможность работы с кэшем TanStack Query, чтобы управлять состоянием на клиенте.

#### Критерии приемки

1. THE System SHALL предоставлять методы для получения query keys (queryKey, pathFilter, queryFilter)
2. WHEN происходит инвалидация, THE System SHALL поддерживать инвалидацию конкретного запроса
3. WHEN происходит инвалидация, THE System SHALL поддерживать инвалидацию всего роутера
4. WHEN происходит инвалидация, THE System SHALL поддерживать инвалидацию всех oRPC запросов
5. THE System SHALL поддерживать prefetchQuery для фоновой загрузки данных
6. THE System SHALL поддерживать fetchQuery для получения данных
7. THE System SHALL поддерживать getQueryData для чтения из кэша
8. THE System SHALL поддерживать setQueryData для записи в кэш

### Требование 9: Миграция оптимистичных обновлений

**User Story:** Как разработчик, я хочу сохранить поддержку оптимистичных обновлений, чтобы обеспечить хороший UX.

#### Критерии приемки

1. WHEN происходит mutation с оптимистичным обновлением, THE System SHALL отменять текущие запросы
2. WHEN происходит mutation с оптимистичным обновлением, THE System SHALL сохранять предыдущее состояние
3. WHEN происходит mutation с оптимистичным обновлением, THE System SHALL обновлять кэш оптимистично
4. WHEN mutation завершается с ошибкой, THE System SHALL откатывать изменения к предыдущему состоянию
5. WHEN mutation завершается, THE System SHALL синхронизировать данные с сервером через invalidateQueries

### Требование 10: Сохранение типобезопасности

**User Story:** Как разработчик, я хочу сохранить полную типобезопасность между клиентом и сервером, чтобы избежать ошибок типизации.

#### Критерии приемки

1. THE System SHALL экспортировать тип AppRouter для использования на клиенте
2. THE System SHALL предоставлять типы RouterInputs для входных данных процедур
3. THE System SHALL предоставлять типы RouterOutputs для выходных данных процедур
4. WHEN клиент вызывает процедуру, THE System SHALL обеспечивать автокомплит для имен процедур
5. WHEN клиент передает данные, THE System SHALL валидировать типы входных параметров на этапе компиляции
6. WHEN клиент получает данные, THE System SHALL обеспечивать корректные типы возвращаемых значений

### Требование 11: Миграция Next.js интеграции

**User Story:** Как разработчик, я хочу сохранить интеграцию с Next.js App Router, чтобы использовать серверные компоненты.

#### Критерии приемки

1. THE System SHALL предоставлять API route handler для обработки oRPC запросов
2. THE System SHALL поддерживать создание серверных хелперов для prefetch в Server Components
3. WHEN происходит prefetch на сервере, THE System SHALL использовать HydrationBoundary для передачи данных клиенту
4. THE System SHALL поддерживать Server Actions для mutations
5. THE System SHALL корректно обрабатывать headers в серверном контексте

### Требование 12: Обратная совместимость и постепенная миграция

**User Story:** Как разработчик, я хочу иметь возможность постепенной миграции, чтобы минимизировать риски.

#### Критерии приемки

1. THE System SHALL поддерживать одновременную работу tRPC и oRPC роутеров
2. WHEN происходит миграция роутера, THE System SHALL сохранять идентичные пути API
3. WHEN происходит миграция роутера, THE System SHALL сохранять идентичные сигнатуры процедур
4. THE System SHALL позволять мигрировать роутеры по одному
5. THE System SHALL обеспечивать возможность отката к tRPC в случае проблем

### Требование 13: Тестирование миграции

**User Story:** Как разработчик, я хочу иметь тесты для валидации миграции, чтобы убедиться в корректности работы.

#### Критерии приемки

1. THE System SHALL сохранять все существующие unit тесты для процедур
2. THE System SHALL сохранять все существующие integration тесты
3. WHEN тесты запускаются, THE System SHALL использовать те же моки и фикстуры
4. THE System SHALL обеспечивать идентичное поведение для всех тестовых сценариев
5. THE System SHALL поддерживать property-based тесты с fast-check

### Требование 14: Документация и примеры

**User Story:** Как разработчик, я хочу иметь примеры использования oRPC, чтобы понимать новые паттерны.

#### Критерии приемки

1. THE System SHALL предоставлять примеры миграции простых процедур (query/mutation)
2. THE System SHALL предоставлять примеры миграции процедур с middleware
3. THE System SHALL предоставлять примеры миграции вложенных роутеров
4. THE System SHALL предоставлять примеры использования на клиенте с TanStack Query
5. THE System SHALL предоставлять примеры оптимистичных обновлений
6. THE System SHALL предоставлять примеры server-side prefetch

### Требование 15: Производительность и оптимизация

**User Story:** Как разработчик, я хочу сохранить или улучшить производительность API, чтобы обеспечить быстрый отклик.

#### Критерии приемки

1. THE System SHALL поддерживать батчинг запросов через httpBatchStreamLink
2. THE System SHALL поддерживать streaming для больших ответов
3. WHEN происходит батчинг, THE System SHALL группировать запросы в один HTTP запрос
4. THE System SHALL сохранять текущие настройки кэширования TanStack Query
5. THE System SHALL обеспечивать время выполнения mutations <500ms (как в текущей системе)
