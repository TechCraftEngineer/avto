# Требования к рефакторингу системы интервью

## Введение

Данный документ описывает требования к рефакторингу приложения `apps/interview` для четкого разделения потоков интервью для разовых заданий (gig) и вакансий (vacancy) с использованием паттерна Strategy. Рефакторинг направлен на улучшение расширяемости системы, интеграцию новых возможностей AI SDK v6 и сохранение общей инфраструктуры.

## Глоссарий

- **System**: Приложение для проведения интервью (`apps/interview`)
- **Strategy**: Паттерн проектирования для инкапсуляции специфичного для типа сущности поведения
- **Gig**: Разовое задание для фрилансеров
- **Vacancy**: Вакансия для постоянного трудоустройства
- **Entity_Type**: Тип сущности интервью (gig, vacancy, project)
- **Interview_Stage**: Стадия интервью (intro, org, tech, wrapup и т.д.)
- **Tool**: Инструмент AI для выполнения специфичных действий
- **Scoring_Rubric**: Схема оценки кандидата
- **AI_SDK**: Vercel AI SDK версии 6
- **Property_Based_Testing**: Тестирование на основе свойств с генерацией случайных входных данных

## Требования

### Требование 1: Архитектура на основе паттерна Strategy

**User Story:** Как разработчик, я хочу использовать паттерн Strategy для разделения логики разных типов интервью, чтобы система была расширяемой и поддерживаемой.

#### Критерии приемки

1. THE System SHALL создать интерфейс `InterviewStrategy` с методами для всех специфичных для типа сущности операций
2. THE System SHALL реализовать `BaseInterviewStrategy` с общей функциональностью для всех типов интервью
3. THE System SHALL реализовать `GigInterviewStrategy` для разовых заданий
4. THE System SHALL реализовать `VacancyInterviewStrategy` для вакансий
5. THE System SHALL создать `InterviewStrategyFactory` для создания стратегий на основе типа сущности
6. WHEN тип сущности неизвестен, THE System SHALL использовать стратегию по умолчанию (vacancy)

### Требование 2: Разделение системных промптов

**User Story:** Как разработчик, я хочу иметь модульную систему промптов, чтобы легко настраивать поведение AI для разных типов интервью.

#### Критерии приемки

1. THE System SHALL создать интерфейс `SystemPromptBuilder` для построения системных промптов
2. THE System SHALL реализовать `BaseSystemPromptBuilder` с общими правилами для всех типов интервью
3. THE System SHALL реализовать `GigSystemPromptBuilder` с правилами специфичными для gig интервью
4. THE System SHALL реализовать `VacancySystemPromptBuilder` с правилами специфичными для vacancy интервью
5. THE System SHALL создать `PromptFactory` для создания промптов на основе типа сущности
6. WHEN строится промпт, THE System SHALL включать инструкции для текущей стадии интервью
7. WHEN это первый ответ, THE System SHALL включать специальные инструкции для приветствия

### Требование 3: Динамическое управление инструментами

**User Story:** Как разработчик, я хочу динамически управлять доступными инструментами AI на основе типа сущности и стадии интервью, чтобы AI использовал только релевантные инструменты.

#### Критерии приемки

1. THE System SHALL создать интерфейс `ToolFactory` для создания наборов инструментов
2. THE System SHALL реализовать `BaseToolFactory` с общими инструментами для всех типов интервью
3. THE System SHALL реализовать `GigToolFactory` с дополнительным инструментом `getInterviewProfile`
4. THE System SHALL реализовать `VacancyToolFactory` использующий только базовые инструменты
5. WHEN создается набор инструментов, THE System SHALL фильтровать инструменты на основе текущей стадии интервью
6. THE System SHALL предоставлять метод `getAvailableTools(stage)` возвращающий список доступных инструментов для стадии
7. THE System SHALL предоставлять метод `isToolAvailable(toolName, stage)` для проверки доступности инструмента

### Требование 4: Определение стадий интервью

**User Story:** Как разработчик, я хочу иметь четкие определения стадий интервью для каждого типа сущности, чтобы контролировать поток интервью.

#### Критерии приемки

1. THE System SHALL создать тип `InterviewStageConfig` с полями: id, order, description, allowedTools, maxQuestions, autoAdvance, entryActions
2. THE System SHALL определить базовые стадии: intro, org, tech, wrapup
3. THE System SHALL определить стадии для gig: intro, profile_review, org, tech, task_approach, wrapup
4. THE System SHALL определить стадии для vacancy: intro, org, tech, motivation, wrapup
5. WHEN стадия имеет autoAdvance = true, THE System SHALL автоматически переходить к следующей стадии после достижения maxQuestions
6. WHEN стадия имеет entryActions, THE System SHALL выполнять указанные действия при входе в стадию

### Требование 5: Схемы оценки кандидатов

**User Story:** Как разработчик, я хочу иметь разные схемы оценки для gig и vacancy интервью, чтобы оценивать кандидатов по релевантным критериям.

#### Критерии приемки

1. THE System SHALL создать `gigScoringSchema` с критериями: strengths_weaknesses, expertise_depth, problem_solving, communication_quality, timeline_feasibility
2. THE System SHALL создать `vacancyScoringSchema` с критериями: completeness, relevance, motivation, communication
3. THE System SHALL включать поле `authenticityPenalty` в обе схемы для учета детекции ботов
4. THE System SHALL включать поле `metadata` с версией схемы, длительностью интервью и количеством вопросов
5. THE System SHALL создать `ScoringFactory` для получения схемы на основе типа сущности
6. WHEN завершается интервью, THE System SHALL использовать `generateObject` из AI SDK для генерации структурированной оценки

### Требование 6: Интеграция AI SDK v6

**User Story:** Как разработчик, я хочу использовать новые возможности AI SDK v6, чтобы улучшить контроль над процессом генерации и телеметрию.

#### Критерии приемки

1. THE System SHALL использовать `activeTools` для динамического управления доступными инструментами на каждой стадии
2. THE System SHALL использовать `prepareStep` для логирования начала каждого шага генерации
3. THE System SHALL использовать `onStepFinish` для отслеживания вызовов инструментов на каждом шаге
4. THE System SHALL использовать `experimental_telemetry` для интеграции с OpenTelemetry
5. THE System SHALL использовать `generateObject` с Zod схемами для генерации структурированных оценок
6. THE System SHALL использовать `toolChoice` для управления выбором инструментов AI
7. THE System SHALL сохранять использование `smoothStream` для плавной потоковой передачи
8. THE System SHALL сохранять использование `stepCountIs(25)` для ограничения количества шагов

### Требование 7: Обновление UI компонентов

**User Story:** Как пользователь, я хочу видеть интерфейс адаптированный под тип интервью, чтобы получать релевантную информацию.

#### Критерии приемки

1. WHEN отображается `InterviewGreeting`, THE System SHALL показывать приветствие специфичное для типа сущности
2. WHEN отображается `InterviewContextCard`, THE System SHALL показывать поля релевантные для типа сущности
3. WHEN тип сущности gig, THE System SHALL отображать: бюджет, дедлайн, срок выполнения
4. WHEN тип сущности vacancy, THE System SHALL отображать: регион, место работы, требуемый опыт
5. THE System SHALL использовать badge с текстом "Разовое задание" для gig
6. THE System SHALL использовать badge с текстом "Вакансия" для vacancy
7. THE System SHALL передавать `entityType` из компонента `InterviewChat` во все дочерние компоненты

### Требование 8: Переходы между стадиями

**User Story:** Как разработчик, я хочу иметь гибкую систему переходов между стадиями, чтобы контролировать поток интервью для разных типов сущностей.

#### Критерии приемки

1. THE System SHALL создать интерфейс `StageTransitionConfig` с полями: from, to, condition, params
2. THE System SHALL реализовать метод `canTransition(from, to, context)` в каждой стратегии
3. WHEN переход для gig из intro в profile_review, THE System SHALL всегда разрешать переход
4. WHEN переход для gig из profile_review в org, THE System SHALL требовать минимум 1 заданный вопрос
5. WHEN переход для vacancy из intro в org, THE System SHALL требовать минимум 1 заданный вопрос
6. WHEN переход для vacancy из org в tech, THE System SHALL требовать минимум 3 заданных вопроса
7. THE System SHALL предоставлять контекст перехода с информацией о заданных вопросах, ответах пользователя и времени в текущей стадии

### Требование 9: Банк вопросов

**User Story:** Как разработчик, я хочу иметь разные наборы вопросов для gig и vacancy интервью, чтобы задавать релевантные вопросы.

#### Критерии приемки

1. THE System SHALL создать интерфейс `QuestionBankConfig` с полями: organizationalDefaults, technicalDefaults, customQuestionsField
2. THE System SHALL определить организационные вопросы для gig фокусирующиеся на опыте с аналогичными задачами, сроках и оплате
3. THE System SHALL определить организационные вопросы для vacancy фокусирующиеся на графике работы, зарплате и дате выхода
4. THE System SHALL реализовать метод `getNextQuestion(questionBank, interviewState)` в каждой стратегии
5. WHEN стадия intro, THE System SHALL возвращать первый вопрос для знакомства
6. WHEN стадия org, THE System SHALL возвращать следующий незаданный организационный вопрос
7. WHEN стадия tech, THE System SHALL возвращать следующий незаданный технический вопрос
8. WHEN стадия wrapup, THE System SHALL возвращать финальный вопрос или null

### Требование 10: Обратная совместимость

**User Story:** Как разработчик, я хочу сохранить обратную совместимость с существующими интервью, чтобы не нарушить работу системы.

#### Критерии приемки

1. THE System SHALL сохранять существующую структуру API эндпоинтов
2. THE System SHALL сохранять существующую схему базы данных для сессий интервью
3. THE System SHALL сохранять существующий формат сообщений в истории чата
4. THE System SHALL сохранять существующую логику контроля доступа
5. THE System SHALL сохранять существующую логику детекции ботов
6. THE System SHALL сохранять существующую логику записи и загрузки голосовых сообщений
7. WHEN тип сущности не определен, THE System SHALL использовать vacancy стратегию по умолчанию

### Требование 11: Типобезопасность

**User Story:** Как разработчик, я хочу иметь полное покрытие TypeScript типами, чтобы избежать ошибок во время выполнения.

#### Критерии приемки

1. THE System SHALL определить branded типы для `GigLike` и `VacancyLike`
2. THE System SHALL определить union тип `SupportedEntityType` = "gig" | "vacancy" | "project"
3. THE System SHALL определить тип `StageId` для всех возможных стадий интервью
4. THE System SHALL использовать Zod схемы для валидации всех входных данных
5. THE System SHALL экспортировать TypeScript типы из всех Zod схем используя `z.infer`
6. THE System SHALL использовать строгие типы для всех параметров методов стратегий
7. THE System SHALL использовать дискриминированные union типы для entity-specific данных

### Требование 12: Расширяемость для новых типов сущностей

**User Story:** Как разработчик, я хочу легко добавлять новые типы сущностей интервью, чтобы система могла расти.

#### Критерии приемки

1. THE System SHALL позволять регистрацию новых стратегий через метод `strategyFactory.register(entityType, strategyFactory)`
2. THE System SHALL проверять поддержку типа сущности через метод `strategyFactory.isSupported(entityType)`
3. WHEN добавляется новый тип сущности, THE System SHALL требовать только реализацию интерфейса `InterviewStrategy`
4. WHEN добавляется новый тип сущности, THE System SHALL автоматически использовать его через фабрику
5. THE System SHALL документировать процесс добавления нового типа сущности
6. THE System SHALL предоставлять примеры реализации стратегий для reference

### Требование 13: Логирование и телеметрия

**User Story:** Как разработчик, я хочу иметь детальное логирование и телеметрию, чтобы отслеживать работу системы и диагностировать проблемы.

#### Критерии приемки

1. THE System SHALL логировать начало каждого шага генерации с номером шага
2. THE System SHALL логировать каждый вызов инструмента с именем инструмента и аргументами
3. THE System SHALL логировать переходы между стадиями с информацией о причине перехода
4. THE System SHALL интегрироваться с OpenTelemetry для трассировки запросов
5. THE System SHALL включать в метаданные трассировки: entityType, vacancyId, gigId, sessionId
6. THE System SHALL логировать ошибки с полным стеком вызовов
7. WHEN происходит ошибка, THE System SHALL включать контекст: тип сущности, стадию, последний вопрос

### Требование 14: Тестирование

**User Story:** Как разработчик, я хочу иметь комплексное тестовое покрытие, чтобы гарантировать корректность работы системы.

#### Критерии приемки

1. THE System SHALL иметь unit тесты для каждой стратегии
2. THE System SHALL иметь unit тесты для каждой фабрики
3. THE System SHALL иметь unit тесты для построителей промптов
4. THE System SHALL иметь integration тесты для полного потока интервью
5. THE System SHALL иметь тесты для проверки переходов между стадиями
6. THE System SHALL иметь тесты для проверки фильтрации инструментов по стадиям
7. THE System SHALL иметь тесты для проверки генерации оценок с использованием Zod схем
8. THE System SHALL иметь минимум 80% покрытие кода тестами

### Требование 15: Документация

**User Story:** Как разработчик, я хочу иметь полную документацию архитектуры, чтобы понимать систему и вносить изменения.

#### Критерии приемки

1. THE System SHALL иметь README файл с описанием архитектуры
2. THE System SHALL иметь диаграммы архитектуры в формате Mermaid
3. THE System SHALL иметь документацию по каждому интерфейсу стратегии
4. THE System SHALL иметь примеры использования для каждой стратегии
5. THE System SHALL иметь руководство по добавлению новых типов сущностей
6. THE System SHALL иметь документацию по миграции существующего кода
7. THE System SHALL иметь комментарии JSDoc для всех публичных методов и интерфейсов
