# Требования к интеграции с ЮКасса

## Введение

Данный документ описывает требования к интеграции платформы с платежной системой ЮКасса (YooKassa) для приема онлайн-платежей от пользователей. Интеграция должна обеспечить создание платежей, обработку уведомлений о статусе платежей и безопасное хранение платежной информации.

## Глоссарий

- **Система** — платформа на базе Next.js с tRPC API
- **ЮКасса** — российская платежная система (YooKassa)
- **Платеж** — транзакция оплаты, созданная через API ЮКасса
- **Webhook** — HTTP-уведомление от ЮКасса о изменении статуса платежа
- **Idempotence_Key** — уникальный ключ для обеспечения идемпотентности запросов
- **Confirmation_URL** — URL страницы оплаты ЮКасса, на которую перенаправляется пользователь
- **Return_URL** — URL, на который пользователь возвращается после оплаты
- **ShopId** — идентификатор магазина в ЮКасса
- **SecretKey** — секретный ключ для аутентификации в API ЮКасса

## Требования

### Требование 1: Создание платежей

**Пользовательская история:** Как пользователь, я хочу инициировать платеж, чтобы оплатить услугу или подписку на платформе.

#### Критерии приемки

1. WHEN пользователь инициирует платеж, THE Система SHALL создать платеж в ЮКасса через API с обязательными параметрами (сумма, валюта, return_url)
2. WHEN платеж создается, THE Система SHALL сгенерировать уникальный Idempotence_Key для обеспечения идемпотентности запроса
3. WHEN ЮКасса возвращает успешный ответ, THE Система SHALL сохранить информацию о платеже в базе данных
4. WHEN ЮКасса возвращает Confirmation_URL, THE Система SHALL перенаправить пользователя на страницу оплаты
5. WHEN создание платежа завершается ошибкой, THE Система SHALL вернуть понятное сообщение об ошибке пользователю

### Требование 2: Аутентификация в API ЮКасса

**Пользовательская история:** Как система, я должна безопасно аутентифицироваться в API ЮКасса, чтобы выполнять операции с платежами.

#### Критерии приемки

1. THE Система SHALL использовать Basic Auth с ShopId и SecretKey для всех запросов к API ЮКасса
2. THE Система SHALL хранить ShopId и SecretKey в переменных окружения
3. THE Система SHALL поддерживать отдельные учетные данные для тестового и продакшн режимов
4. WHEN учетные данные отсутствуют или некорректны, THE Система SHALL вернуть ошибку конфигурации

### Требование 3: Обработка webhook-уведомлений

**Пользовательская история:** Как система, я должна получать уведомления от ЮКасса о статусе платежей, чтобы обновлять информацию в базе данных.

#### Критерии приемки

1. WHEN ЮКасса отправляет webhook-уведомление, THE Система SHALL принять и обработать его
2. WHEN webhook получен, THE Система SHALL проверить подлинность уведомления
3. WHEN статус платежа изменился на "succeeded", THE Система SHALL обновить статус платежа в базе данных на "успешно завершен"
4. WHEN статус платежа изменился на "canceled", THE Система SHALL обновить статус платежа в базе данных на "отменен"
5. WHEN webhook обработан успешно, THE Система SHALL вернуть HTTP 200 OK
6. WHEN обработка webhook завершается ошибкой, THE Система SHALL логировать ошибку и вернуть HTTP 500

### Требование 4: Проверка статуса платежа

**Пользовательская история:** Как система, я должна иметь возможность проверить текущий статус платежа, чтобы синхронизировать данные с ЮКасса.

#### Критерии приемки

1. WHEN требуется проверить статус платежа, THE Система SHALL выполнить GET-запрос к API ЮКасса с идентификатором платежа
2. WHEN ЮКасса возвращает статус платежа, THE Система SHALL обновить информацию в базе данных
3. WHEN платеж не найден в ЮКасса, THE Система SHALL вернуть ошибку "платеж не найден"

### Требование 5: Хранение данных о платежах

**Пользовательская история:** Как система, я должна хранить информацию о всех платежах, чтобы отслеживать историю транзакций.

#### Критерии приемки

1. THE Система SHALL сохранять следующие данные о платеже: идентификатор ЮКасса, сумма, валюта, статус, дата создания, дата обновления
2. THE Система SHALL связывать платеж с пользователем, который его инициировал
3. THE Система SHALL сохранять описание платежа (назначение платежа)
4. WHEN платеж создается, THE Система SHALL установить начальный статус "pending"
5. THE Система SHALL хранить Idempotence_Key для каждого платежа

### Требование 6: Идемпотентность операций

**Пользовательская история:** Как система, я должна обеспечить идемпотентность создания платежей, чтобы избежать дублирования при повторных запросах.

#### Критерии приемки

1. WHEN создается платеж, THE Система SHALL сгенерировать уникальный Idempotence_Key на основе UUID
2. WHEN повторный запрос с тем же Idempotence_Key отправляется в течение 24 часов, THE ЮКасса SHALL вернуть информацию о ранее созданном платеже
3. THE Система SHALL сохранять Idempotence_Key в базе данных для отслеживания

### Требование 7: Валидация данных платежа

**Пользовательская история:** Как система, я должна валидировать данные платежа перед отправкой в ЮКасса, чтобы предотвратить ошибки.

#### Критерии приемки

1. WHEN создается платеж, THE Система SHALL проверить, что сумма больше нуля
2. WHEN создается платеж, THE Система SHALL проверить, что валюта является допустимой (RUB)
3. WHEN создается платеж, THE Система SHALL проверить, что return_url является валидным URL
4. WHEN описание платежа превышает 128 символов, THE Система SHALL обрезать его до 128 символов
5. WHEN данные невалидны, THE Система SHALL вернуть ошибку валидации с описанием проблемы

### Требование 8: Обработка возврата пользователя

**Пользовательская история:** Как пользователь, я хочу видеть результат оплаты после возврата с страницы ЮКасса, чтобы понимать, успешна ли транзакция.

#### Критерии приемки

1. WHEN пользователь возвращается на Return_URL, THE Система SHALL получить идентификатор платежа из параметров URL
2. WHEN пользователь возвращается, THE Система SHALL проверить актуальный статус платежа через API ЮКасса
3. WHEN платеж успешен, THE Система SHALL отобразить страницу успешной оплаты
4. WHEN платеж отменен или ожидает подтверждения, THE Система SHALL отобразить соответствующее сообщение
5. WHEN идентификатор платежа отсутствует или некорректен, THE Система SHALL отобразить страницу ошибки

### Требование 9: Логирование и мониторинг

**Пользовательская история:** Как разработчик, я хочу иметь логи всех операций с платежами, чтобы отслеживать проблемы и анализировать транзакции.

#### Критерии приемки

1. THE Система SHALL логировать все запросы к API ЮКасса (без секретных данных)
2. THE Система SHALL логировать все полученные webhook-уведомления
3. THE Система SHALL логировать все ошибки при работе с платежами
4. WHEN происходит ошибка аутентификации, THE Система SHALL логировать детали ошибки
5. THE Система SHALL логировать изменения статуса платежей

### Требование 10: Типобезопасность и валидация схем

**Пользовательская история:** Как разработчик, я хочу использовать типобезопасные схемы для всех операций с платежами, чтобы предотвратить ошибки типизации.

#### Критерии приемки

1. THE Система SHALL использовать Zod v4 для валидации всех входящих и исходящих данных
2. THE Система SHALL определить TypeScript типы для всех сущностей платежей
3. THE Система SHALL использовать Zod схемы для валидации ответов от API ЮКасса
4. THE Система SHALL использовать Zod схемы для валидации webhook-уведомлений
5. WHEN данные не соответствуют схеме, THE Система SHALL вернуть детальную ошибку валидации
