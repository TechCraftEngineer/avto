# Requirements Document

## Introduction

Система назначения ответственного рекрутера для вакансий позволяет управлять распределением вакансий между рекрутерами в рамках workspace организации. Каждая вакансия должна иметь назначенного ответственного рекрутера, который отвечает за работу с этой вакансией.

## Glossary

- **System**: Система управления назначением ответственных рекрутеров
- **Vacancy**: Вакансия в системе рекрутинга
- **Recruiter**: Пользователь с ролью рекрутера в workspace
- **Owner**: Ответственный рекрутер, назначенный на вакансию
- **Workspace**: Рабочее пространство в рамках организации
- **Organization**: Организация, содержащая workspace и пользователей

## Requirements

### Requirement 1: Назначение ответственного рекрутера

**User Story:** Как менеджер по найму, я хочу назначить ответственного рекрутера для вакансии, чтобы было понятно, кто отвечает за работу с этой вакансией.

#### Acceptance Criteria

1. WHEN пользователь назначает рекрутера на вакансию, THE System SHALL сохранить связь между вакансией и рекрутером
2. WHEN назначается ответственный рекрутер, THE System SHALL проверить, что рекрутер является членом того же workspace, что и вакансия
3. WHEN назначается ответственный рекрутер, THE System SHALL проверить, что рекрутер имеет активный статус в workspace
4. WHEN вакансия создается без указания ответственного, THE System SHALL разрешить создание вакансии с пустым полем owner
5. THE System SHALL возвращать информацию об ответственном рекрутере при запросе данных вакансии

### Requirement 2: Изменение ответственного рекрутера

**User Story:** Как менеджер по найму, я хочу изменить ответственного рекрутера для вакансии, чтобы перераспределить нагрузку или заменить рекрутера.

#### Acceptance Criteria

1. WHEN пользователь изменяет ответственного рекрутера, THE System SHALL обновить связь между вакансией и новым рекрутером
2. WHEN изменяется ответственный рекрутер, THE System SHALL проверить, что новый рекрутер является членом того же workspace
3. WHEN изменяется ответственный рекрутер, THE System SHALL сохранить timestamp изменения
4. WHEN ответственный рекрутер удаляется (устанавливается null), THE System SHALL разрешить удаление связи
5. THE System SHALL возвращать обновленную информацию о вакансии после изменения owner

### Requirement 3: Отображение информации об ответственном рекрутере

**User Story:** Как пользователь системы, я хочу видеть информацию об ответственном рекрутере для вакансии, чтобы знать, к кому обращаться по вопросам.

#### Acceptance Criteria

1. WHEN запрашиваются данные вакансии, THE System SHALL включать полную информацию об ответственном рекрутере (имя, email, аватар)
2. WHEN ответственный рекрутер не назначен, THE System SHALL возвращать null для поля owner
3. THE System SHALL возвращать информацию об owner в формате, совместимом с существующими моделями пользователей
4. WHEN запрашивается список вакансий, THE System SHALL включать информацию об owner для каждой вакансии

### Requirement 4: Фильтрация вакансий по ответственному рекрутеру

**User Story:** Как рекрутер, я хочу видеть только свои вакансии, чтобы сосредоточиться на своей работе.

#### Acceptance Criteria

1. WHEN пользователь применяет фильтр по ответственному рекрутеру, THE System SHALL возвращать только вакансии с указанным owner
2. WHEN фильтр по owner не указан, THE System SHALL возвращать все вакансии workspace
3. THE System SHALL поддерживать фильтрацию по нескольким ответственным рекрутерам одновременно
4. WHEN применяется фильтр "без ответственного", THE System SHALL возвращать вакансии с owner равным null
5. THE System SHALL комбинировать фильтр по owner с другими существующими фильтрами вакансий

### Requirement 5: Валидация данных

**User Story:** Как система, я хочу валидировать входные данные, чтобы обеспечить целостность данных.

#### Acceptance Criteria

1. WHEN получен запрос на назначение owner, THE System SHALL проверить, что vacancy ID является валидным UUID
2. WHEN получен запрос на назначение owner, THE System SHALL проверить, что recruiter ID является валидным UUID
3. WHEN получен невалидный vacancy ID, THE System SHALL возвращать ошибку с кодом 400
4. WHEN получен ID рекрутера из другого workspace, THE System SHALL возвращать ошибку с кодом 403
5. WHEN получен ID несуществующей вакансии, THE System SHALL возвращать ошибку с кодом 404

### Requirement 6: Права доступа

**User Story:** Как администратор системы, я хочу контролировать, кто может назначать ответственных рекрутеров, чтобы обеспечить безопасность данных.

#### Acceptance Criteria

1. WHEN пользователь пытается назначить owner, THE System SHALL проверить, что пользователь имеет права на изменение вакансии
2. WHEN пользователь не имеет прав, THE System SHALL возвращать ошибку с кодом 403
3. THE System SHALL разрешать назначение owner только пользователям с ролями admin или manager в workspace
4. WHEN рекрутер пытается назначить owner на чужую вакансию, THE System SHALL проверить права доступа
5. THE System SHALL разрешать рекрутеру просматривать информацию об owner для всех вакансий в workspace
